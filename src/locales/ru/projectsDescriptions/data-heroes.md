# Notification Policy Service

Backend-сервис для управления пользовательскими настройками уведомлений и правилами их доставки.

Проект представляет собой модульный API-сервис, который отвечает за хранение предпочтений пользователей, применение глобальных политик и принятие решения о возможности отправки уведомлений.

Основная задача сервиса - предоставить единый механизм проверки перед отправкой уведомления:

* разрешено ли отправлять уведомление;
* какой канал доставки использовать;
* запрещена ли отправка глобальными правилами;
* есть ли пользовательские ограничения.

---

## Возможности

### Управление пользователями

Сервис поддерживает создание пользователей и хранение их настроек уведомлений.

Основные данные пользователя:

* идентификатор;
* регион;
* часовой пояс;
* временные ограничения для уведомлений;
* персональные настройки каналов доставки.

Пример пользователя:

```json
{
  "id": 1,
  "region": "EU",
  "timezone": "Europe/Berlin",
  "quietHours": {
    "from": "22:00",
    "to": "08:00"
  }
}
```

---

## Notification Policy Engine

Главная часть проекта - механизм оценки политики уведомлений.

Перед отправкой сообщения система проверяет:

1. Существуют ли глобальные ограничения.
2. Разрешен ли канал доставки.
3. Разрешен ли конкретный тип уведомления.
4. Не находится ли пользователь в периоде ограничения отправки.

Пример сценария:

> Пользователь находится в регионе EU.
> Глобальная политика запрещает маркетинговые SMS для EU.
> Пользователь включил маркетинговые уведомления.
>
> Итог: уведомление не отправляется, так как глобальная политика имеет более высокий приоритет.

# Основные технические решения

## Модульная архитектура

Каждый доменный модуль содержит собственную бизнес-логику:

* HTTP слой;
* сервисы;
* DTO;
* схемы валидации;
* Swagger документацию.

Такой подход позволяет масштабировать проект без смешивания ответственности между компонентами.

---

## Type-safe валидация через Zod

Все входящие данные проходят runtime-проверку.

Zod используется для:

* проверки API payload;
* генерации типов TypeScript;
* повторного использования схем между слоями приложения.

Пример:

```ts
const evaluateSchema = z.object({
  userId: z.number(),
  channel: z.enum([
    'email',
    'sms',
    'push'
  ]),
  notificationType: z.string()
})
```

---

## Database слой

Для работы с PostgreSQL используется:

* Drizzle ORM;
* Drizzle Kit;
* SQL миграции.

Преимущества подхода:

* полный контроль над SQL;
* типизированные запросы;
* прозрачная структура базы данных.

---

## Request Context

Для хранения контекста запроса используется `AsyncLocalStorage`.

Это позволяет получать данные текущего запроса в сервисах без необходимости прокидывать дополнительные параметры через каждый метод.

Пример:

```ts
const context = requestContext.get()

console.log(context.user)
```

Используется для:

* логирования;
* трассировки запросов;
* доступа к данным пользователя.

# API

## Создание пользователя

### POST `/users/create`

Создает нового пользователя.

Request:

```json
{
  "region": "EU",
  "timezone": "Europe/Berlin"
}
```

Response:

```json
{
  "id": 1
}
```

---

## Получение настроек пользователя

### GET `/users/:id/preferences`

Возвращает текущие настройки уведомлений пользователя.

Пример:

```
GET /users/1/preferences
```

Response:

```json
{
  "email": true,
  "sms": false,
  "push": true,
  "quietHours": {
    "from": "22:00",
    "to": "08:00"
  }
}
```

---

## Обновление настроек пользователя

### POST `/users/:id/preferences`

Обновляет пользовательские настройки.

Request:

```json
{
  "sms": false,
  "marketing": false
}
```

Response:

```json
{
  "success": true
}
```

---

# Проверка возможности отправки уведомления

## POST `/evaluate`

Основной endpoint сервиса.

Используется перед отправкой уведомления.

Request:

```json
{
  "userId": 1,
  "channel": "sms",
  "notificationType": "marketing"
}
```

Response:

```json
{
  "allowed": false,
  "reason": "Blocked by global policy"
}
```
